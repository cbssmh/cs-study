# Dynamic Programming (DP)

## Definition
Dynamic Programming solves problems by:
- dividing them into smaller subproblems
- storing previously computed results
- reusing optimal subproblem solutions

Applicable when:
- overlapping subproblems exist
- optimal substructure exists

---

## Core Idea

> Solve small problems first → build larger optimal solutions from them.

Instead of recomputing:
- save results in a table (DP table)
- or memoize recursive calls

Key question:

> “How should the DP state/table be defined?”

---

## 0/1 Knapsack

## Definition
Choose items with:
- weight `wi`
- value `vi`

Goal:
- maximize total value
- without exceeding capacity `W`

Each item:
- either taken (`1`)
- or not taken (`0`)

---

## Core Idea

Define:

`DP[i][j]`
= maximum value using items `0 ~ i`
with capacity `j`

For each item:
- do not take it
- take it

Choose the better option.

---

## Process

For item `i`:

### Case 1: Do not take item
`DP[i-1][j]`

### Case 2: Take item
`DP[i-1][j-wi] + vi`

Transition:

```text
DP[i][j] = max(
    DP[i-1][j],
    DP[i-1][j-wi] + vi
)
```

---

## Implementation

- 2D table commonly used
- Iterate:
  - items
  - capacities

Important:
- capacity must remain valid
- previous row represents smaller subproblem

---

## Key Insight

> Optimal solution of a large problem can be built from optimal solutions of smaller capacities.

---

## Complexity

- Time: `O(NW)`
- Space: `O(NW)` or optimized `O(W)`

---

## Notes

### Very Important
0/1 Knapsack is **NOT** Greedy.

Reason:
- locally best choice may not produce global optimum.

---

# Longest Common Subsequence (LCS)

## Definition

Find the longest subsequence appearing in both strings.

Subsequence:
- characters remain in order
- not necessarily contiguous

---

## Core Idea

Compare:
- last character of `X`
- last character of `Y`

Two cases:
- characters match
- characters differ

---

## Process

If characters match:

```text
LCS[i][j] = LCS[i-1][j-1] + 1
```

If different:

```text
LCS[i][j] = max(
    LCS[i-1][j],
    LCS[i][j-1]
)
```

---

## Implementation

DP table:
- rows → string X
- columns → string Y

Fill table left-to-right, top-to-bottom.

---

## Key Insight

> Matching characters extend previous optimal subsequences.

---

## Complexity

- Time: `O(NM)`
- Space: `O(NM)`

---

## Notes

Common mistake:
- confusing substring vs subsequence

Substring:
- continuous

Subsequence:
- order only

---

# Triangle DP Problem

## Definition

Move from top to bottom of a triangle.

At each step:
- move diagonally left/right downward

Goal:
- maximize total sum

---

## Core Idea

Each position can be reached from:
- upper-left
- upper-right

Choose larger accumulated value.

---

## Process

```text
DP[i][j] = max(
    DP[i-1][j-1],
    DP[i-1][j]
) + triangle[i][j]
```

---

## Implementation

- Use 2D DP table
- Boundary handling is important

---

## Key Insight

> Convert path selection into local maximum accumulation.

---

# Tiling Problem (2×N)

## Definition

Count ways to tile:
- `2×N` rectangle
using:
- `1×2`
- `2×1` tiles

---

## Core Idea

Focus on:
- how the last section is filled

Two cases:
- vertical tile
- two horizontal tiles

---

## Process

```text
DP[n] = DP[n-1] + DP[n-2]
```

---

## Implementation

Base cases:
- `DP[1] = 1`
- `DP[2] = 2`

Then build upward.

---

## Key Insight

> Final tile placement determines recurrence relation.

---

## Complexity

- Time: `O(N)`
- Space: `O(N)` or `O(1)`

---

# Downhill Path Problem

## Definition

Move:
- up/down/left/right

Condition:
- only move to smaller values

Goal:
- count total valid paths

---

## Core Idea

Simple DP ordering fails because:
- dependencies come from multiple directions

Use:
- DFS + Memoization

---

## Process

DFS explores neighbors:
- if lower height

Memo table stores:
- number of paths from each cell

Base case:
- destination reached → return 1

---

## Implementation

### DFS + Memoization

Pseudo-flow:
1. visit cell
2. explore valid neighbors
3. sum returned path counts
4. save result
5. reuse cached value later

---

## Key Insight

> Memoization eliminates repeated DFS exploration.

---

## Complexity

- Near `O(V+E)` effectively
- Much faster than brute force DFS

---

## Notes

### Very Important
This is:
- recursive DP
- memoized search

NOT standard iterative table DP.

---

# Greedy Algorithm

## Definition

Greedy algorithm:
- chooses the locally optimal choice
- never revises previous decisions

Assumption:
- local optimum → global optimum

---

## Core Idea

At every step:
- choose the best immediate option

No backtracking.

---

## Process

1. Evaluate candidates
2. Pick best current choice
3. Lock decision
4. Repeat

---

## Key Insight

> Greedy works only when local optimal choices guarantee global optimality.

---

## Complexity

Usually efficient:
- low time complexity
- simple implementation

---

## Notes

### Very Important
Not every optimization problem is Greedy-solvable.

Counterexamples exist frequently.

---

# Fractional Knapsack

## Definition

Variant of Knapsack:
- items may be divided

---

## Core Idea

Choose highest:
- value per weight ratio

First.

---

## Process

Compute:

```text
ratio_i = vi / wi
```

Then:
1. sort descending
2. take fully if possible
3. otherwise take fraction

---

## Implementation

Typically:
- sorting + linear scan

---

## Key Insight

> Since items are divisible, taking highest value density first is always optimal.

---

## Complexity

- Sorting dominates:
`O(N log N)`

---

## Notes

### Very Important
Fractional Knapsack → Greedy works

0/1 Knapsack → Greedy fails

---

# Coin Change (Greedy Version)

## Definition

Goal:
- make amount `A`
- using minimum number of coins

---

## Core Idea

Repeatedly choose:
- largest possible coin

---

## Process

1. pick largest valid coin
2. subtract value
3. repeat

---

## Implementation

Works for:
- canonical coin systems
- e.g. Korean currency

---

## Key Insight

> Larger denominations reduce coin count faster.

---

## Notes

### Very Important
Greedy coin change does NOT always work.

Counterexample:
- coins `{1,3,4}`
- target `6`

Greedy:
`4 + 1 + 1 = 3 coins`

Optimal:
`3 + 3 = 2 coins`

---

# Prim Algorithm

## Definition

Greedy algorithm for:
- Minimum Spanning Tree (MST)

---

## Core Idea

Expand tree by:
- selecting minimum edge connecting
  - visited set
  - unvisited set

---

## Process

1. start from one vertex
2. choose smallest connecting edge
3. add new vertex
4. repeat

---

## Implementation

Usually:
- Priority Queue
- adjacency list

---

## Key Insight

> Always extending with minimum crossing edge preserves MST optimality.

---

## Complexity

- Heap version:
`O(E log V)`

---

# Kruskal Algorithm

## Definition

Greedy MST algorithm using:
- edge sorting

---

## Core Idea

Pick:
- globally smallest edge

If:
- cycle not formed

→ include edge

---

## Process

1. sort edges
2. iterate smallest first
3. use Union-Find
4. skip cycles

---

## Implementation

### Very Important
Union-Find required for cycle detection.

---

## Key Insight

> Smallest non-cycling edges safely construct MST.

---

## Complexity

- Sorting dominates:
`O(E log E)`

---

# Dijkstra Algorithm

## Definition

Shortest path algorithm for:
- non-negative edge weights

---

## Core Idea

Repeatedly finalize:
- currently closest unvisited vertex

---

## Process

1. initialize distances
2. pick smallest unvisited node
3. relax neighbors
4. repeat

---

## Implementation

### Relaxation

```text
dist[v] = min(
    dist[v],
    dist[u] + w(u,v)
)
```

Priority Queue commonly used.

---

## Key Insight

> Once a node is finalized, its shortest distance is guaranteed.

---

## Complexity

- Heap implementation:
`O(E log V)`

---

## Notes

### Very Important
Fails with:
- negative edge weights

Use:
- Bellman-Ford instead

---

# Meeting Room Scheduling

## Definition

Choose maximum number of non-overlapping meetings.

Each meeting:
- start time
- end time

---

## Core Idea

Always choose:
- earliest finishing meeting

Reason:
- leaves maximum remaining space

---

## Process

1. sort by finish time
2. select first meeting
3. repeatedly select compatible earliest-ending meeting

---

## Implementation

Sorting is essential.

---

## Key Insight

> Earliest finishing time maximizes future scheduling opportunities.

---

## Complexity

- Sorting:
`O(N log N)`

---

# Sequence Reordering Minimization Problem

## Definition

Rearrange array `A`
to minimize:

```text
S = Σ(Ai × Bi)
```

`B` remains fixed.

---

## Core Idea

Match:
- largest values in `B`
with
- smallest values in `A`

---

## Process

1. sort `A` ascending
2. pair with descending `B`

---

## Key Insight

> Opposite pairing minimizes total weighted contribution.

---

## Notes

Related to:
- Rearrangement Inequality
